# Independent verification: script-classification cluster

Second-pass review of `terms/alphabet.md`, `abjad.md`, `abugida.md`, `alphasyllabary.md`, `syllabary.md`, `featural-alphabet.md`, `logographic.md`, `logosyllabary.md`, `ideographic.md`, `pictographic.md`, plus the one-sentence typology edit to `language-terms/writing-systems-and-scripts/script.md`. Reviewed against `main` at commit `6baf328` (pulled fresh; note below on repo sync). No files were edited.

**Repo-sync note:** the folder connected to this session lagged behind `origin/main` by several commits during this review (missing commit `6baf328`, "add 'scripts of this type' sampler tables to the six real script-type entries," and five earlier "review X to current standard" commits). All findings below are against the fully-pulled `main`, confirmed via `git log` in a fresh clone check. If your local working copy still shows the older content, pull before applying any fixes.

## Findings

**1. (should-fix) Six new sampler tables carry no source citations.** Commit `6baf328` added "scripts of this type" tables to `alphabet.md`, `abjad.md`, `abugida.md`, `syllabary.md`, `featural-alphabet.md`, and `logographic.md` (e.g., abugida.md now lists Devanagari, Bengali-Assamese, Tamil, Tibetan, Thai, and Ethiopic as abugidas). None of the ~25 new rows carry a `<sup>n</sup>` marker, and no entry's Sources list gained a row-level citation. House spec requires the opposite: "Script samples are sourced too... Verify sampler tables per script, not by family... the table is a living subset... its row sources are numbered first" (`glossary-canonical-entry-format.md`, Sourcing and writing rules). I independently confirmed the classifications themselves are correct (Thai as abugida, Persian/Urdu as abjad, Ethiopic as the etymological abugida, Armenian/Georgian as true alphabets) via Unicode Ch. 18/20 and general reference, but the entries don't cite that anywhere.
   Fix: add a numbered source per row (or per table, if one source covers the whole set, e.g., ScriptSource or the Unicode script-block chapter), placed first in each entry's Sources list, ahead of the existing prose citations, and renumber the `<sup>` markers in body prose accordingly.

**2. (should-fix, six entries) `summary` does not equal the Definition's first sentence.** The automated gate (`check_entry.py`) and a manual diff both confirm mismatches in `alphasyllabary.md`, `featural-alphabet.md`, `logographic.md`, `logosyllabary.md`, `ideographic.md`, and `pictographic.md`. This is a mandatory checklist item ("summary equals the Definition's FIRST sentence," `glossary-canonical-entry-format.md`).
   - `alphasyllabary.md`: summary says "...abugida, a script where each consonant..." and "...changed by added marks"; Definition says "...abugida: a script where each consonant..." and "...changed by adding marks." Punctuation and verb form drift.
   - `featural-alphabet.md`: summary truncates the actual first sentence, dropping ", so related sounds get visually related letters."
   - `logographic.md`: summary drops the parenthetical "(morphemes)."
   - `logosyllabary.md`: summary drops "(also morphosyllabary)" and paraphrases "at the same time, for a spoken syllable" as "also for a syllable."
   - `ideographic.md` and `pictographic.md`: both summaries stop at the first semicolon, dropping the Unicode-example clause ("Unicode's example is ♻/✈, which denotes...") that is part of the same first sentence in the body.
   Fix: make each `summary` field an exact copy of the Definition's first sentence (up to the first sentence-ending period), or split the Definition's first sentence at the semicolon/comma so the two can match exactly.

**3. (nit) `abugida.md` fails the automated no-dash gate.** "Bengali–Assamese" uses an en dash (in the new table and in Related terms and mentions), which `check_entry.py` flags. House rule bans em dashes and tells authors to convert Cowork's en-dashes in link labels to colons; a compound script name isn't quite either case, but the simplest fix that satisfies the gate and reads the same is a plain hyphen: "Bengali-Assamese" (also the more common spelling in the sources I checked).

**4. (should-fix) `abugida.md` mentions "alphasyllabary" without linking it or listing it.** The Definition says "An abugida is also called an alphasyllabary," but `alphasyllabary.md` is a real, defined entry (an exact synonym, per its own Definition: "The two terms are interchangeable") and it isn't linked inline or present in "Related terms and mentions." This is exactly the kind of taxonomic-spine edge the completeness checklist calls out.
   Fix: link on first mention, `[alphasyllabary](alphasyllabary.md)`, and add Alphasyllabary to Related terms and mentions.

**5. (should-fix) `abugida.md` mentions Font, Glyph, and Typeface in prose without listing them.** "Why it matters in design systems" says "...picking a **typeface** with the right **glyphs**" and "...a **font** passing a character-grid check..." None of Font, Glyph, or Typeface appear in Related terms and mentions, though all three are defined terms (`terms/font.md`, `terms/glyph.md`, `terms/typeface.md`).
   Fix: add Font, Glyph, and Typeface to Related terms and mentions (alphabetized alongside the existing list). The scan's other flags for this file (Character, Mark, Language) read as generic-English false positives on this pass; worth a quick human glance but not clear misses.

**6. (should-fix) `alphabet.md` mentions Letter spacing and Typeface in prose without listing them.** "Why it matters in design systems" says "...case, sort order, and **letter spacing** all vary..." and "...before you commit to a **typeface** or a layout." Both `terms/letter-spacing.md` and `terms/typeface.md` exist.
   Fix: add Letter spacing and Typeface to Related terms and mentions.

**7. (should-fix) Three-way inconsistency on where Canadian Aboriginal Syllabics belongs.** `featural-alphabet.md`'s new sampler table lists Canadian Aboriginal Syllabics as an example of a featural script, and its prose calls it "a featural abugida." The parent MOC (`language-terms/writing-systems-and-scripts/README.md`) files Canadian Aboriginal Syllabics under "Abugida scripts" (not under "Featural scripts," which lists only Hangul). And `abugida.md`'s own new sampler table omits Canadian Aboriginal Syllabics entirely, despite the MOC placing it there. House rule: "Match the parent MOC... If it should change, change the parent too, do not diverge silently."
   Fix: pick one home (or list it in both tables, since the entry's own language is "a featural abugida," i.e., genuinely both) and update whichever of the three documents is now the odd one out.

**8. (should-fix, answers your flagged item 1) `featural-alphabet.md`'s HTML review comment can be resolved, not just left flagged.** The comment says the Hangul "tongue and lips" articulatory account, the "featural abugida" label for Canadian Aboriginal Syllabics, and the King Sejong/15th-century attribution are stated without a fetched primary. I independently searched for sourcing on all three and found them well-documented and easy to source: the articulatory account and the 1443/1446 King Sejong attribution trace to the *Hunminjeongeum Haerye* (the 1446 commentary, a UNESCO Memory of the World document, translated and discussed on Wikipedia's "Origin of Hangul" and in standard Korean-linguistics references); "featural abugida" for Canadian Aboriginal Syllabics is an established characterization independent of this glossary (e.g., discussed on Wikipedia's "Abugida" article). None of this contradicts what's in the entry now; it's accurate, just uncited.
   Fix: cite a specific source for each (e.g., Wikipedia "Origin of Hangul" or a Hunminjeongeum Haerye translation for the articulatory account and Sejong attribution; Wikipedia "Abugida" for the CAS characterization), then remove the HTML comment per the "no internal voice, uncertainty resolved before publish" rule.

**9. (worth flagging, not a hard error) The primary term name may itself be the confusable.** `featural-alphabet.md`'s flagship, most-cited example is Hangul, which the entry's own cited source calls a "featural **syllabic** script," not an alphabet. The parent MOC's "Scripts to explore" section headings it "Featural scripts:" (generic), not "Featural alphabets," while the MOC's own "Script classification terms" bullet list still says "Featural alphabet." Given the house rule that definitions must be script-neutral and hold across script types, and that "featural script" is already an alias in this entry's frontmatter, consider whether "Featural script" should be the primary term (with "Featural alphabet" as the alias) rather than the reverse. This is a naming/purpose-gate observation for the human reviewer to weigh, not something I'd block on; the current Definition prose is already careful about the Hangul nuance.

**10. (nit) Two unmarked quote truncations.** These don't change meaning but don't follow the letter of "any '...' must be verbatim":
   - `logographic.md` source 3 and `ideographic.md` source 2 both quote DeFrancis: "...as do all real writing systems." The source sentence continues "...despite their diverse techniques and differing effectiveness in accomplishing the task." (https://www.pinyin.info/readings/texts/ideographic_myth.html). Not flagged with an ellipsis.
   - `ideographic.md` source 1 quotes the Unicode Glossary Ideograph entry: "...which denotes the concept of recycling." The raw page continues "...by a series of bent arrows." (https://www.unicode.org/glossary/#ideograph, confirmed via direct fetch). Also unmarked.
   Separately, both this quote and `pictographic.md` source 1 silently convert the source's em dash ("—for example") to a spaced hyphen (" - for example"), presumably to satisfy the house no-em-dash rule; worth doing consistently and consciously rather than as a side effect, but I wouldn't block on it.
   Fix: add "..." at each drop point, or quote through to the sentence's actual end.

**11. (nit) `logosyllabary.md` cites an adjacent Unicode definition instead of its own dedicated one.** It cites the "Logogram" entry's sense (2), "A generic term for the unit of writing of a logosyllabic writing system." Unicode's glossary also has a dedicated, more directly on-point entry: "Logosyllabary. A writing system in which the units are used primarily to write words and/or morphemes of words, with some subsidiary usage to represent just syllabic sounds. The best example is the Han script." (https://www.unicode.org/glossary/#logosyllabary, confirmed via direct fetch). The current citation isn't wrong, but the more precise source exists and isn't used.
   Fix: cite/link `#logosyllabary` instead of, or alongside, `#logogram`.

**12. (nit) Attribution could be more precise.** `logographic.md` source 2 attributes "They are logographic (each character represents a word, not just a sound) characters that developed from pictographic and ideographic principles" to "The Unicode Standard, Version 16.0, Chapter 18." Confirmed verbatim on the page, but that sentence is itself a blockquote of the Japanese dictionary *Koujien* (1983) that Chapter 18 reproduces, not Unicode's own authorial voice. Substance and URL are fine; consider noting "as quoted in the Unicode Standard" for precision.

**13. (nit) `entry_type: pointer` is undocumented.** `alphasyllabary.md` and `logosyllabary.md` both use `entry_type: pointer`, a value not defined anywhere in `glossary-canonical-entry-format.md` (which only documents `entry_type: overview` for the 7 topic terms). Used identically by two entries, so it reads as an intentional pattern for alias/redirect-style entries, not a typo, but the canonical format doc should say so if it's meant to stick.

**14. (judgment call, answers your flagged item 2) `ideographic.md`'s demotion to core looks defensible but worth a second look.** Commit `61c5178` demoted it from deep to core. The current "Why it matters" beat already carries the misuse-correction (Han is logographic, not ideographic) and the practitioner guidance (reach for "logographic" in docs/specs/UI copy) that Common mistake / In practice would otherwise repeat, so the demotion doesn't look like padding was lost. But the canonical format doc explicitly reserves deep status for "the confusable pairs" (`glossary-canonical-entry-format.md`), and ideographic-vs-logographic-vs-pictographic is exactly that kind of pair. Recommend the human reviewer confirm this was a deliberate overlap call, since the depth checklist asks for the one-line reason to be stated in the review record and I didn't find one on this file.

**15. (verified, no issue, answers your flagged item 3) `pictographic.md`'s retained-deep status is earned.** Commit `011cf8f` kept it deep and tightened it. Common mistake (cultural non-universality of pictograms) and In practice (icon+label pairing, user testing with the real audience, don't overclaim universality) are each distinct from "Why it matters" and from each other, and each connects to a real design-system decision (accessibility, localization, icon systems), not filler.

## What checked out clean

- **Marker discipline:** all `<sup>n</sup>` markers ascend in order of first appearance and balance 1:1 with each entry's Sources list, across all 10 entries plus `script.md`. Confirmed by direct read and by `check_entry.py`.
- **Quote accuracy:** every `<sup>` claim I checked against a live fetch of its cited source matched (word for word, aside from the ellipsis notes above). Fetched and checked: Unicode Glossary (alphabet, abjad, abugida, kana, logogram, ideograph, pictograph, script definitions), W3C "An Introduction to Writing Systems & Unicode," UAX #9 (bidi), Microsoft's Devanagari OpenType doc, Unicode 16.0 Ch. 18 (East Asia) and Ch. 20 (Americas), DeFrancis's "The Ideographic Myth," W3C "Quote marks," CLDR "Script Names," and r12a's Gurmukhi/Punjabi orthography notes.
- **Confusables cleanly distinguished:** alphabet vs. abjad vs. abugida, syllabary vs. abugida, and logographic vs. logosyllabary vs. ideographic vs. pictographic are all correctly and consistently drawn, and Han characters are correctly and consistently labeled logographic/logosyllabary, never ideographic, backed by both DeFrancis and Unicode's own "conventional cover term" caveat in Ch. 18.
- **No Latin- or horizontal-default-as-universal claims** anywhere in the cluster; RTL (Arabic, Hebrew), vertical (Han), and non-Latin directional behavior are treated as first-class throughout.
- **Rendered script samples** (мир, مرحبا, क/कि, あ/い, 山, ♻, ✈, नमस्ते, 你好) all render and mean what each entry states, cross-checked against fetched primaries and general reference, not assumed from memory.
- **Links:** every `(slug.md)` target across all 11 files resolves to a real file. No orphans found.
- **Formatting:** heading levels (`## Definition` only, `###` for every other beat) and the `***` divider immediately before "Related terms and mentions" are correct in all 11 files. No em dashes found in body prose (the en dash in `abugida.md`'s "Bengali–Assamese" is the one dash-family issue, item 3). No bare "sign" usage found anywhere in the cluster. "Letter" is used only in genuinely alphabet/abjad-specific contexts; no generic-unit misuse found.
- **Alphabetization:** "Related terms and mentions" is correctly alphabetized in all 11 files.
- **`script.md`'s one-sentence typology edit** (commit `c0a7fb9`, "align script.md typology list") is accurate: parenthesizing alphasyllabary as another name for abugida and logosyllabary as the more precise term for logographic matches both `alphasyllabary.md` and `logosyllabary.md`'s own Definitions, word for word in substance. No issue.

## Verdict

Not yet clean: needs another pass. The core content is factually sound and the confusable-pair distinctions are the strongest part of this cluster, but the six new sampler tables (item 1) are a real, systemic house-style gap (uncited factual claims, the exact thing this review method exists to catch), and the six summary/Definition mismatches (item 2) fail the project's own automated gate. Items 4-6 (missing backlinks) and item 7 (the Canadian Aboriginal Syllabics three-way inconsistency) are quick, concrete fixes. Items 8-15 are lower-stakes or judgment calls for the human reviewer.
